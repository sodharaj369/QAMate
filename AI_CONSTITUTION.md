# QAMate AI Constitution
**Operational Charter for AI-First QA Reasoning Workspaces**

---

## 🏛️ Principal Directive
QAMate is **AI-First but Human-Resilient**. Offline *automated reasoning heuristics* are removed. Semantic analysis and strategy planning require active AI reasoning. However, the AI is treated as **one evidence source among many**, not the center of the application. 

The ultimate output is a unified **QAMate Mental Model** compiled by the core engine combining AI observations, rule templates, project memory, workspace intelligence, and the **Project DNA**.

If AI providers are blocked or offline, QAMate enters **Manual Analysis Mode** (visualizing the **AI Disconnected Mode** UI state), degrading into a structured manual workspace where the human QA Lead manually maps the system model and strategy.

---

## 1. Core Architectural Pillars

### The Evidence-Based Reasoning Flow
All cognitive pipelines must flow through the structured evidence collection pipeline:
```
Requirement ➔ Evidence Extraction ➔ [AI Observations / Knowledge / Workspace Intelligence / Project DNA] ➔ System Model ➔ QA Mental Model ➔ Test Strategy ➔ QA Deliverables
```
* **System Model Ownership**: The System Model belongs entirely to QAMate. AI only contributes observations to the Evidence Extraction phase.
* **QA Mental Model Gate**: The QA Mental Model is the heart of QAMate, capturing the active facts, assumptions, inferences, and unknowns.

### Permanent Architecture Rule: Structured Data Over Documents
Every capability and engine in QAMate must produce and store **structured JSON data**, not raw documents. Markdown files, Excel sheets, PDF reports, and script files are strictly transient **views and export formats** rendered by the export engine. QAMate never stores files in raw document formats internally.

### The Project DNA Store (Visible & Transparent)
The **Project DNA Store** represents the permanent settings of the workspace and is fully visible and editable in the UI. It contains: tech stack, coding standards, business vocabulary, domain glossary, system architecture, testing standards, team preferences, known limitations, existing suites, integration landscape, and reusable components.

### The AI Request Optimizer (Redesigned TokenOptimizer)
The **AI Request Optimizer** manages payload compilation, static context caching, delta prompting, prompt chunking, provider-specific prompt formatting, context prioritization, Project DNA injection, and token budgeting. It is strictly forbidden from stripping code comments, rewriting user input, or changing input meanings.

### The QA Value Engine (Unique Selling Proposition)
All raw test case recommendations generated by AI must pass through the **QA Value Engine**. The engine is responsible for duplicate case removal, parameterization of validation checks, merging similar objectives, and coverage optimization. It compiles test cases into high-value sets.

### The QA Review Engine
The compliance gate auditor responsible for evaluating weak expected results, missing negative assertions, missing boundaries (when relevant), checking traceability matrices, and verifying style and playbook compliance.

### The QA Recommendation Engine (Platform-Wide)
A reusable platform-wide reasoning component that evaluates any active workspace state (Requirement, System Model, Mental Model, or Strategy) against the Project DNA to output structured recommendations:
`Requirement/State ➔ Recommendation ➔ Reason ➔ Industry Practice ➔ Accept/Ignore`

### The Change Intelligence Engine & Change History
The engine must compute requirement modifications across sprints, tracking **Story Evolution** (existing regression tests, new cases, and regression impact) and recording a **Change History Log** showing *why* a Strategy or Test Case changed:
`Requirement changed ➔ System Model updated ➔ Risk recalculated ➔ Strategy expanded ➔ Cases regenerated`

## 2. Division of Responsibilities

### AI Responsibilities
* **Observation**: Reporting semantic observations, system actors, interfaces, and implicit risks.
* **Advisory Recommendations**: Suggesting testing focuses and exclusions based on context.
* **Drafting Gaps**: Identifying requirement omissions and drafting clarification cards.
* **Synthesizing Test Logic**: Generating manual steps, automation code skeletons, and API payloads.

### QAMate Core Responsibilities
* **Orchestration**: Guiding the session state through the six stepper outcomes.
* **Schema Validation**: Enforcing strict JSON schema contracts on AI responses.
* **Compliance Verification**: Performing checks by executing local static checks and computing rule coverage matrices. (AI only contributes audit observations).
* **Operation Contracts Enforcement**: Enforcing guarantees (Semantic Integrity, Logical Exclusivity) on all workspace operations.
* **Manual Analysis Mode**: Degrading into a structured manual editor when AI is offline.

### Human Responsibilities
* **DNA Administration**: Defining and maintaining the Project DNA file directly in the UI.
* **Assumptions Verification**: Reviewing and confirming the AI-extracted assumptions list.
* **Direct GUI Editing**: Modifying objectives, scope, and out-of-scope zones directly in GUI input fields. (No raw markdown editing is exposed in the UI).
* **Final Approval**: Reviewing, swiping, accepting/ignoring recommendations, and authorizing delivery.

---

## 3. UI/UX & Interaction Principles

### One Layout, Hidable Panels
The workspace features **one unified layout** to prevent documentation duplication. Panels and views (e.g., manual steps, code skeletons) are hidden, collapsed, or expanded based on active user preferences, rather than redesigning the layout for different personas.

### Confidence for Communication
Confidence metrics (Understanding, Strategy, Generation) are computed and displayed to communicate readiness to the user. However, **confidence scores are never used to programmatically gate or decide workflow transitions**.

### Structured Edit (No Exposed Markdown)
Objectives, scope, and out-of-scope zone edits must occur through structured GUI input fields. The engine stores the data as JSON internally and exports it as Markdown, but **never exposes raw markdown editing to the user**.

---

## 4. Operation Contracts & Guarantees

Every workspace operation must satisfy its corresponding contract:

1. **Understand (System Modeling)**
   * *Guarantee*: *Structural Integrity*—The source requirement text is never modified or corrupted.
2. **Prepare (Clarification Gating)**
   * *Guarantee*: *Semantic Traceability*—Every clarification question maps directly to an active gap in the System Model.
3. **Plan (Strategy Formulation)**
   * *Guarantee*: *Logical Exclusivity*—Every test suite marked as "Excluded" must feature a user-approved justification.
4. **Generate (Artifact Synthesis)**
   * *Guarantee*: *No Placeholders*—Generated manual steps or code scripts contain zero `"TODO"` or dummy assertions.
5. **Review (Compliance Auditing)**
   * *Guarantee*: *Semantic Integrity*—The generated tests verify the exact logic boundaries of the requirements without introducing new assumptions.

---

## 5. Traceability, Change Tracking & Governance

### Explainability (Reasoning Trace)
Every recommendation generated by the engine must display a complete trace mapping:
`Evidence ➔ Reasoning ➔ Decision ➔ Confidence ➔ Recommendation`

### Knowledge Approval
All lessons learned, user corrections, and bug patterns captured during a session are stored as pending entries. They must be presented to the user for review before being committed to the project's permanent memory database.

### Learning Approval
The project's memory belongs to the human team. The AI must never silently write to the permanent knowledge base. Dynamic modifications, rules extensions, or templates additions require explicit human authorization to prevent database contamination.
